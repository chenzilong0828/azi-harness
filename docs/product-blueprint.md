# AI Coding Harness 产品构想

## 产品机会

目标用户是使用多个 AI 编程工具的研发团队。核心问题不是模型不会写代码，而是团队的
需求、职责、权限、规则、测试和交付标准没有被稳定地传递给 Agent。

产品定位：将团队研发流程编译成各类 AI 编辑器可消费的项目级协议。

## 三个视角的功能构想

### 产品经理

1. 一键安装团队研发工作流。
2. 按阶段生成 PRD、Issue、测试和 Review 交付物。
3. 团队模板版本锁定及升级记录。
4. 流程完成度和 AI 越权行为审计。
5. 不同项目类型的预设 Profile。

### 产品设计

1. 交互式初始化向导。
2. `doctor` 健康检查与可执行修复建议。
3. 当前阶段、阻塞项和下一步的清晰展示。
4. 修改规则前的差异预览。
5. 人工审批点和风险提示。

### 软件工程

1. 编辑器适配器层，将统一协议映射到 Codex、Claude Code、Cursor 等格式。
2. 可组合、可测试、带版本的 Skill 包。
3. 工作流有限状态机和阶段门禁。
4. JSON Schema 校验 PRD、Issue 和测试产物。
5. 本地优先的 Registry、签名和供应链安全机制。

## 优先级最高的五项

1. **统一项目协议安装器**
   最快交付核心价值。假设：团队愿意将规则以 Markdown/JSON 维护在仓库。

2. **编辑器适配器**
   形成差异化。假设：不同工具之间存在足够稳定的项目配置映射。

3. **工作流状态机**
   防止 Agent 跳过需求和测试。假设：多数团队接受默认流程，并允许按项目裁剪。

4. **Doctor 与产物校验**
   让规范从“文档建议”变成可验证约束。假设：关键产物可以定义机器可读的最低标准。

5. **版本化 Skill Registry**
   支持团队复用和演进。假设：Skill 能保持低耦合，并有清晰的输入输出契约。

## MVP 非目标

- 不训练或托管大模型。
- 不取代 Codex、Claude Code、Cursor。
- 不自动执行生产发布。
- 不把所有团队流程固化成唯一模板。
- 不在首版建设中心化云服务。
